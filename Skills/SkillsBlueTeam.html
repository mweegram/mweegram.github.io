Log Reading and Analysis

Working as a Cyber Security Analyst, I have read a lot of logs and have a good eye for spotting behaviour that deviates from what is expected and classes as 'business as usual'. I can read both Windows and Linux logs (important as the main two OS families) and interpret what has happened whether that be a password change or a file download.

I have also seen a lot of Cloud Service logs, whether that be AWS or Azure and have dealt with logs from both. Using the information provided to me from the logs I am able to make intelligent and informed decisions to reach an optimal conclusion whether that be interacting with the user/developer responsible, or when neccessary, escalating to Incident Co-Ordinator.

Due to my degree borrowing its structure from the Cisco CCNA I have a very solid background in networks, most specifically protocols such as DNS, TCP, HTTP, LDAP and DHCP. I have read all types of protocol based logs and diagnosed malicious behaviour in each of them in the past. For example, I am able to very easily spot attempts at attacks such as Log4J exploitation in Web and LDAP logs.

Incident Response

I have been apart of real security incidents such as suspected security breaches and followed the threat throughout it's life cycle (from detection to remediation). As a SOC analyst I have personally detected many live threats. One important skill I picked up as part of this role is curiosity, I now always work out a bigger picture of not only what I can see in the tickets and what that means for the incident, but what else is going on behind the scenes such as related attacks, campaigns and other connections from Source and Destination.

After escalating threats to the neccessary parties, I would first build a picture of what has happened (download of malicious file, credentials phished or information leaked). Next, I would work out how this happened (phishing email or policy violation). I would then investigate the compromised host or account and see if there is any other behaviour that is out of the ordinary. Now I would isolate or contain the affected device or account (Lock the account or device through service provider like Active Directory). If possible I would eradicate any threats through Anti-Malware systems or complete wipes of systems. In some cases I would then request the device for detailed forensics on the incident.

I have worked on many incidents such as active malware, phishing links clicked (not always with credentials leaked but had to prove this using logs) and live hacking machines (Kali) on the network and dealing with that appropriately. As part of this role I spent a lot of time verifying if a targetted system was susceptible to the attack that was attempted via software inventory and searching vulnerable software and CVEs.

In this process I had to deal with different types of events some of which provided different information than others.

Intrusion Detection/Prevention Systems

During my time working as an Incident Responder (temporarily), I was able to become very familiar working with Intrusion Detection Systems (IDS) and Intrusion Prevention Systems (IPS). These systems show attacks that have managed to breach external firewalls and have managed to make it far enough into the network. The rule is that if an incident shows on IDS you should assume it was successful. However the purpose of IPS is that it has prevented the attack.

These systems feed their information into the SIEM and generate alerts into tickets, using the provided information I would then determine what the intended target was (in some cases this would show as a load balancer and I would have to work out the devices in the load balanced pool) and determine if the threat could have been successful, if so I would then escalate the alert to the Incident manager and wait for my next steps. Due to the experience I have gained in this area I am now confident dealing with information and alerts that come from IDS/IPS devices and can use them as a really useful information source.

Firewall/Web Traffic Logs

I have used Firewalls and Web Traffic logs as an aide in the process of investigation and triage of intial threats, I have used them to work out what a user or suspicious IP address was doing around the time that an alert was triggered or for use as data enrichment in further investigations.

I was able to check whether or not packets had made it past the perimeter firewall and whether or not further traffic has been blocked or denied. Another good use of these skills was to review whether a user was able to successfully send a POST request after clicking a malicious phishing link. Web logs also came in handy as enrichment data to work out whether a user visting a suspicious/strange (scams etc) website coincide with their web history.

One important area that merges the two topics of this section is Web Application Firewalls (WAF) which are firewalls sitting further up the OSI model at Layer 7 (Application Layer). WAFs monitor traffic going towards web applications and have the ability to prevent attacks and exploits if they are spotted by dropping the packets. I have been able to use WAF logs in investigations to work out if an attack was successful and follow the correct procedures from that point onwards to remediate threat or provide rationale for closing a ticket.
